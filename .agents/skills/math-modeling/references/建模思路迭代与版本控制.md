# 建模思路迭代与版本控制

本参考文件用于数学建模任务中出现多版建模思路、多轮报告修改、需要 Git 留痕时使用。

## 一、多版建模思路对比流程

当用户提供新的建模思路，并要求与当前 `题目分析报告.md` 对比时，先在聊天区完成对比，不要直接把对比过程写入报告。

对比应按以下层次展开：

1. 总体框架：判断两版思路对三问递进关系的把握是否一致。
2. 问题本质：逐题比较问题类型判断是否准确，如 OP、TOP、VRP、POMDP、RH-MPC 等。
3. 数学模型：比较风险、收益、约束、目标函数和状态变量的表达。
4. 求解算法：比较算法可行性、复杂度、竞赛实现成本和可视化效果。
5. 敏感性与鲁棒性：比较是否覆盖参数敏感性、战损重分配、通信中断等实验。
6. 论文表达：比较哪种思路更容易写出清晰模型、图表和结果分析。

聊天区输出应明确说明：

- 哪一部分沿用当前报告；
- 哪一部分吸收用户新思路；
- 哪一部分只作为备选或对照实验；
- 为什么这样选择。

## 二、写入分析报告的规则

经用户确认后，写入 `题目分析报告.md` 时只保留最终融合后的建模方案。

可以写入：

- 最终模型类型；
- 最终变量定义；
- 最终目标函数和约束；
- 最终求解流程；
- 最终敏感性和鲁棒性实验；
- 最终可视化和论文展示建议。

不要写入：

- “当前思路 vs 新思路”的过程性比较；
- “哪版更好”的评判文字；
- “综合比较后”“取舍理由如下”等对话痕迹；
- 用户原文中的未整理段落；
- 与最终模型无关的讨论。

## 三、建模报告迭代建议

报告迭代时优先遵循以下结构：

1. 先更新模型主线，再更新公式。
2. 再更新求解流程，确保算法与模型一致。
3. 最后更新术语表，补充新增模型、算法和指标。
4. 每次写入后搜索报告中是否残留“对比、取舍、哪版更好”等过程性词语。
5. 若用户明确要求对比内容保留在文档中，才单独增加“方案比较”章节。

## 四、Git 版本控制流程

每完成一轮稳定建模修改，应使用 Git 保存版本。优先使用提交信息记录变更，不在工作区创建冗余版本说明文件。

### 提交原则

1. 提交信息使用简体中文。
2. 提交标题简短说明主题。
3. 提交正文说明本轮具体改动。
4. 不提交 `.agents/`、Office 临时锁文件、dump/json/zip 中间文件。
5. 不把无关删除或临时文件混入建模提交。

推荐提交格式：

```bash
git commit -m "完善建模思路v2" -m "补充问题1风险阈值与期望收益求解流程；补充问题2综合效能评价与鲁棒性分析；补充问题3事件驱动滚动重规划。"
```

### 标签原则

重要版本使用简练中文标签：

```bash
git tag "建模思路v1" <commit>
git tag "建模思路v2" <commit>
git tag "求解代码v1" <commit>
git tag "论文初稿v1" <commit>
```

标签命名应短、中文、能表示阶段，不使用英文标签。

### 查看与比较历史版本

查看某版报告：

```bash
git show 建模思路v1:建模分析/题目分析报告.md
```

导出某版报告：

```bash
git show 建模思路v1:建模分析/题目分析报告.md > 建模分析/题目分析报告_v1.md
```

比较两个版本：

```bash
git diff 建模思路v1 建模思路v2 -- 建模分析/题目分析报告.md
```

查看提交历史：

```bash
git log --oneline --decorate
```

## 五、交付前检查

完成一次建模思路迭代后，至少检查：

1. `题目分析报告.md` 是否只包含最终方案；
2. `术语表格.md` 是否包含新增术语；
3. Git 暂存区是否只包含本轮应提交文件；
4. 标签是否为简练中文；
5. 是否还有未处理的无关临时文件。
